# 신고(Report) 시스템 기획 및 구현 계획

사용자가 부적절한 콘텐츠나 유저를 신고하고, 관리자가 이를 검토하여 조치를 취할 수 있는 종합적인 신고 시스템을 구축합니다.

## 기획 사양

### 1. 신고 대상 및 카테고리
- **대상**: 유저(USER), 게시글(POST), 댓글(COMMENT)
- **카테고리**: [부적절한 내용], [비난 및 욕설], [스팸/홍보], [허위 사실], [기타]

### 2. 데이터베이스 모델 설계 (Prisma)
- **`Report` 모델**:
    - `targetType`: 신고 대상 유형 (USER, POST, COMMENT)
    - `targetId`: 신고 대상의 고유 ID
    - `reporterId`: 신고를 한 유저 ID
    - `reportedUserId`: 신고를 당한 유저 ID (조회 최적화용)
    - `category`: 신고 카테고리
    - `reason`: 상세 사유 (선택 사항)
    - `status`: 처리 상태 (PENDING, RESOLVED, REJECTED)
    - `adminAction`: 관리자 조치 결과 (WARNING, BAN_7, BAN_PERMANENT, NONE)
    - `createdAt`: 신고 일시

### 3. 주요 기능 및 API 로직

#### [NEW] 신고 접수 API (`POST /api/reports`)
- 사용자가 특정 대상(유저/글/댓글)을 신고.
- **누적 체크**: 신고 당한 유저의 `Report` 테이블 내 누적 횟수를 실시간으로 계산하여 응답에 포함하거나 관리자 알림 트리거.

#### [NEW] 관리자 신고 관리 API (`GET /api/reports`, `PATCH /api/reports/:id`)
- **목록 조회**: 접수된 신고를 최신순/상태별로 조회.
- **조치 실행**: 관리자가 신고를 검토하고 조치를 선택.
    - `경고 발송`: 유저에게 경고 알림 전송.
    - `7일 정지`: 유저의 상태를 일시 중지.
    - `영구 제재`: 유저를 영구 밴 처리.
    - `허위신고/반려`: 신고 상태를 반려로 변경.

## 구현 단계

1.  **데이터베이스**: `schema.prisma`에 `Report` 모델 추가 및 마이그레이션.
2.  **컨트롤러**: `report.controller.ts`를 생성하여 신고 접수 및 관리 로직 구현.
3.  **라우트**: `report.routes.ts` 엔드포인트 연결.
4.  **문서화**: Swagger UI에 신고 API 명세 업데이트.

## 검증 계획
- 동일 유저가 동일 대상을 중복 신고할 수 없도록 방지 로직 확인.
- 신고 시 해당 유저의 전체 신고 누적 횟수가 정확히 조회되는지 확인.
- 관리자 조치 시 실제 유저 테이블의 상태(membership 등)에 반영되는지 검증.
