# 고도화된 콘텐츠 필터링(Content Filtering) 구현 계획

사용자의 피드백을 반영하여, 원본 데이터를 보존하면서도 커뮤니티에는 정제된 콘텐츠만 노출되도록 시스템을 설계합니다.

## 1. 필터링 정책 및 로직 고도화

### 1.1 데이터 이중 저장 (Original vs Masked)
- **대상**: `Post`, `Comment` 모델
- **로직**:
    - `content`: 사용자가 입력한 **원본 데이터** 저장. (마이페이지/작성자 본인에게만 노출)
    - `maskedContent`: 비속어가 `***`로 치환된 **필터링 데이터** 저장. (일반 유저 및 목록에 노출)
- **이점**: 작성자는 자신의 글을 온전히 볼 수 있으며, 타인에게는 정제된 내용만 제공됩니다.

### 1.2 정교한 링크/도메인 감지 (Anti-Ad)
- **문제**: `http` 없이 `naver.com`, `ex.com` 식으로 올리는 경우 감지 필요.
- **해결**: TLD(Top-Level Domain) 패턴을 포함한 정규표현식 및 `linkify-it`과 같은 전문 라이브러리를 사용하여 도메인 형태의 텍스트를 전수 감지.
- **조치**: 감지된 모든 도메인은 `[보안상 차단된 링크]`로 마스킹 처리합니다. (현재 예외 허용 도메인 없음)

### 1.3 비속어 관리 체계
- **개발자 기본 제공**: 범용적인 한국어 비속어 사전(`src/utils/badwords.json`) 우선 탑장.
- **관리자 실시간 관리**: `BannedWord` 테이블을 통해 관리자가 운영 중에도 금지어를 추가/삭제 가능하도록 구현.

## 2. 데이터베이스 스키마 수정 (Prisma)
- **`Post` / `Comment`**: `maskedContent` (String) 필드 추가.
- **`BannedWord`**: 금지어 정보를 담는 신규 모델 추가.

## 3. 필터링 유틸리티 (`src/utils/filter.ts`)
- `maskProfanity(text)`: 비속어를 `***`로 변환.
- `maskLinks(text)`: 도메인 및 URL 형식을 `[차단된 링크]`로 변환.
- `processContent(text)`: 위 두 과정을 거쳐 최종 `maskedContent` 반환.

## 4. 구현 단계
1.  **DB 업데이트**: Prisma 스키마 수정 및 필드 추가.
2.  **유틸리티 개발**: 전문 라이브러리를 활용한 필터링 함수 구현.
3.  **컨트롤러 연동**: 게시글/댓글 생성/수정 시 `maskedContent` 자동 생성 로직 삽입.
4.  **목록 조회 로직 수정**: API 요청 유저가 작성자가 아닐 경우 `maskedContent`를 반환하도록 처리.
